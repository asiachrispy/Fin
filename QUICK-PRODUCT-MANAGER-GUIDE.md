# 快速产品经理技能 - 使用指南

## 技能概述

**技能名称**：`quick-product-manager`

**核心特点**：
- ✅ **快速响应**：基于用户简短描述，主动推断需求
- ✅ **方案导向**：一次性给出完整方案，而非逐个问题
- ✅ **业务理解**：熟悉常见业务场景和最佳实践
- ✅ **最小交互**：力争 1-2 轮对话完成需求定义

## 常见问题

### Q1：如果 AI 推断错误怎么办？

**A**：在方案确认时选择修改：

```
请确认：
2. ⚠️ 需要修改：不需要导出功能，也不需要权限管理
```

AI 会快速调整方案。

---

### Q2：如何让 AI 生成更准确的方案？

**A**：提供更多信息，但不需要完整的需求文档：

```
好的描述：
"我需要个销售报表，展示最近30天数据，需要趋势图、
分类对比、详细表格，要能导出，不需要权限控制"

不太好的描述：
"我需要个报表"（太模糊）
```

---

### Q3：方案确认后如何生成文档？

**A**：AI 会自动生成：

```
你：✅ 方案正确
AI：好的，正在生成 Product-Spec.md...
   生成完成！
   文件位置：Product-Spec.md
   包含内容：功能规划、页面结构、数据结构、技术栈
```

---

### Q4：与 product-spec-builder 的区别？

| 维度 | product-spec-builder | quick-product-manager |
|------|-------------------|---------------------|
| 交互方式 | 逐个问题追问 | 一次性输出方案 |
| 对话轮次 | 5-15 轮 | 1-2 轮 |
| 耗时 | 15-30 分钟 | 2-5 分钟 |
| 适用场景 | 复杂需求、需要精确 | 快速原型、MVP 定义 |
| 用户感受 | 系统化但繁琐 | 快速高效 |

**建议**：
- 简单/中等复杂度需求：使用 `quick-product-manager`
- 高复杂度/关键业务需求：使用 `product-spec-builder`

## 最佳实践

### 1. 描述需求时提供关键信息

```
好的描述：
"我需要个销售报表，展示最近30天数据，
核心指标：销售额、订单量、转化率，
需要趋势图和详细表格，支持导出"

包含信息：
- 时间范围
- 核心指标
- 图表类型
- 交互需求
```

### 2. 明确不要什么

```
"我需要个销售报表，
不需要：权限管理、审批流程、复杂的多维分析"
```

### 3. 明确技术栈偏好（如有）

```
"使用 Next.js + shadcn/ui + Recharts"
"不需要后端，前端硬编码数据即可"
```

### 4. 说明业务目标

```
"这个报表的目标是帮助运营团队快速了解销售状况，
用于日常决策，不需要深度分析功能"
```

## 总结

| 技能 | 适用场景 | 对话轮次 | 耗时 |
|------|---------|---------|------|
| **quick-product-manager** | 快速原型、MVP 定义 | 1-2 轮 | 2-5 分钟 |
| **product-spec-builder** | 复杂需求、精确定义 | 5-15 轮 | 15-30 分钟 |

**推荐策略**：
- 数据团队快速创建报表原型：优先使用 `quick-product-manager`
- 需求非常复杂或有严格合规要求：使用 `product-spec-builder`
